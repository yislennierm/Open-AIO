# Architecture

```text
+-------------------+      POST telemetry       +-------------------+
| Windows PC agent  | -------------------------> | FastAPI server    |
|                   |                            |                   |
| - psutil stats    |                            | - app mapping     |
| - foreground exe  |                            | - asset manifests |
| - no icons/titles |                            | - safe file serve |
+-------------------+                            +---------+---------+
                                                           ^
                                                           |
                                      GET state/assets     |
                                                           |
                                                   +-------+-------+
                                                   | ESP32 display |
                                                   |               |
                                                   | - polls state |
                                                   | - caches logo |
                                                   | - renders TFT |
                                                   +---------------+
```

## PC Agent

The PC agent is intentionally small. It reads the foreground process executable basename through platform-specific APIs/tools and gathers CPU/RAM telemetry through `psutil`. Temperatures are best-effort because many operating systems do not expose them through `psutil`.

The agent posts only:

- active process basename
- CPU and RAM load
- optional CPU/GPU temperatures
- optional GPU load

It never sends window titles, icons, screenshots, command lines, or paths.

Foreground detection support:

- Windows: Win32 foreground window PID APIs.
- Linux/X11: `xdotool getactivewindow getwindowpid`.
- Linux/Sway Wayland: `swaymsg` plus `jq`.
- macOS: `osascript` frontmost process name.

## Server

The FastAPI server authenticates every API request with `X-API-Key`. It stores the latest telemetry in memory for the MVP and maps normalized process basenames to stable app IDs.

The server owns display assets. Asset folders are under `server/assets/apps/{app_id}` and are addressed by known app IDs only. Each app folder should contain a `manifest.json` generated by the preprocessing script.

Unknown processes map to `default`.

## ESP32

The ESP32 connects to Wi-Fi, mounts LittleFS, and polls:

```text
GET /api/v1/device/{device_id}/state
```

When the returned `asset_hash` differs from the cached hash, the firmware downloads `asset_url`, verifies SHA-256, and saves the RGB565 file plus metadata to LittleFS.

## Cache Flow

1. Server returns state JSON with asset URL and SHA-256.
2. ESP32 compares server hash with `/asset.hash`.
3. If hashes differ, ESP32 downloads the asset with a max byte cap.
4. ESP32 computes SHA-256 while saving to a temporary file.
5. If the hash matches, the temporary file replaces the cached logo.
6. If anything fails, the previous cached logo remains in use.
